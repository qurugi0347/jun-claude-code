## 리뷰 실행

위 "Codex 가용성" 출력값에 따라 분기한다.

### A. codex_available

`/codex:review` 슬래시 커맨드를 우선 사용하여 코드 리뷰를 수행한다.

**Codex 위임 프롬프트 구성**:
- 리뷰 대상: 위 "구현 코드 변경 내역" 범위 (`main...HEAD` 또는 `HEAD~5..HEAD`)
- Plan 컨텍스트: `.claude/plan/plan.md`, `.claude/plan/context.md`
- 적용할 6개 기본 리뷰 관점 (아래 "리뷰 관점" 섹션 전체 전달)
- 추가 지시: **구현의 목적, 도메인, 기술 스택에 따라 필요한 관점을 자유롭게 추가하여 리뷰한다**
- 산출물 형식: 본 문서 하단의 "출력 형식"을 그대로 따른다 (요약/Critical/Warning/긍정 평가/미구현 항목, 표 컬럼: 파일:라인/관점/내용/수정 제안)

심층 리뷰 옵션:
- 기본 리뷰로 부족하다고 판단되거나 보안·아키텍처 변경이 큰 경우 `/codex:adversarial-review`를 추가 실행하여 보강한다.

Codex 응답 수신 후 Claude가 수행할 작업:
1. Codex 리뷰 결과를 본 문서의 "출력 형식"에 맞춰 정규화
2. 표 컬럼(파일:라인/관점/내용/수정 제안) 검증 및 누락분 보강
3. 사용자에게 최종 리뷰 결과 제시

### B. codex_unavailable

`code-reviewer` agent를 호출하여 아래 관점에서 구현 코드를 검토하고 Critical/Warning으로 분류한다.

## 리뷰 관점

아래는 기본 리뷰 관점이다. **구현의 목적, 도메인, 기술 스택에 따라 추가로 필요한 관점이 있다면 자유롭게 추가하여 리뷰한다.**

### 1. 목적 부합성
- 구현이 context.md 사용자 요청을 충족하는가?
- plan.md 설계 결정대로 구현되었는가?
- plan에 있으나 누락된 기능이 있는가?

### 2. 오버엔지니어링 검토
- 요구사항 대비 코드 복잡도가 적절한가?
- 사용하지 않을 추상화/인터페이스가 있는가?
- 더 단순한 구현 방법이 있는가?

### 3. 성능 관점
- N+1 쿼리, 불필요한 루프 등 성능 이슈가 있는가?
- 대량 데이터 처리 시 문제가 될 수 있는 부분이 있는가?
- 캐싱 적용 가능한 부분이 있는가?

### 4. 사용자 Flow 관점
- 에러 상태, 로딩 상태, 빈 상태가 처리되었는가?
- 사용자에게 적절한 피드백이 제공되는가?
- 예외 케이스가 처리되었는가?

### 5. 유지보수 관점
- 코드가 기존 프로젝트 패턴과 일관성이 있는가?
- 단일 책임 원칙이 지켜졌는가?
- 테스트 가능한 구조인가?

### 6. 보안 관점
- 사용자 입력이 적절히 검증되는가?
- 인증/권한 처리가 올바른가?
- 민감 정보가 노출되지 않는가?

## 긍정 평가
